# 社交平台发布执行包

这份文档不是“泛泛的推广建议”，而是给你直接拿去发内容的执行包。目标是两件事：

1. 为项目引流
2. 为产品收集真实求职者反馈

## 一、平台 API 现状判断

### GitHub

- 最适合当前阶段持续公开构建过程
- 可用仓库 README、Issues、Discussions、Releases、项目截图做冷启动
- 官方文档：
  - https://docs.github.com/en/rest/issues
  - https://docs.github.com/graphql/guides/using-the-graphql-api-for-discussions

### B 站

- 目前最值得继续走官方接口申请的国内平台
- 公开文档里可以看到开发者入驻、OAuth、令牌换取等能力
- 适合以后做视频化分发或自动稿件流转
- 参考：
  - https://bilibili.apifox.cn/
  - https://bilibili.apifox.cn/doc-885661
  - https://bilibili.apifox.cn/api-23688085

### 小红书

- 有开放平台、开发者注册、资质审核和内容工具类目规则
- 但当前不建议假设“普通创作者通用发笔记接口”已经可以直接稳定使用
- 适合先按人工发帖流程运营，同时跟踪开放平台能力
- 参考：
  - https://xiaohongshu.apifox.cn/doc-2810914
  - https://xiaohongshu.apifox.cn/doc-2811130

### V2EX

- 官方 API 2.0 已公开，适合读数据、管理提醒、查看主题等
- 当前公开文档里没有“通用创建主题”接口作为主要依赖
- 因此更适合作为人工发帖渠道，不适合作为自动化分发主阵地
- 参考：
  - https://www.v2ex.com/help/api

### 知乎 / 掘金

- 这轮没有核实到适合作为“普通创作者自动发文”的稳定官方发布 API
- 建议先按人工发布内容包执行

## 二、统一内容策略

每个平台都不要发“空泛宣传”。统一内容结构建议如下：

1. 痛点开头
2. 一个真实错误判断或真实缺口
3. 系统如何给出更可信的解释
4. 结果页截图
5. GitHub 链接或试用入口

## 三、统一图片素材清单

建议先做 6 张标准图，所有平台复用：

1. 封面图
2. 首页上传页截图
3. 结果页顶部总览截图
4. 要求级证据卡片截图
5. 面试承接截图
6. 公开资料补充截图

图片制作建议：

- 宽图适合知乎、掘金、GitHub README
- 竖图适合小红书
- 每张图只突出一个重点，不要把整页全塞进去
- 对隐私信息做马赛克或直接用样例简历

## 四、各平台可直接发的内容包

### 1. GitHub README 顶部简介

推荐一句话：

> 一个面向中文求职场景的简历诊断系统，强调 ATS 可读性、JD 对齐、证据卡片和面试承接，而不是只给一个不透明分数。

README 顶部配图建议：

- 首页截图
- 结果页总览截图
- 一张要求级证据卡片截图

### 2. GitHub Release 文案

标题：

`v0.x - 增加公开资料补充、任务页、用户自带大模型预留和隐私检查`

正文结构：

1. 这版新增了什么
2. 为什么要加
3. 怎么本地验证
4. 还缺什么
5. 欢迎反馈什么问题

### 3. GitHub 招募 Issue

标题建议：

`[Beta 招募] 想找 5 位真实求职者，帮我验证简历诊断是否真的有用`

正文建议：

我在做一个中文求职场景下的简历诊断系统，重点不是生成花哨结论，而是尽量把 ATS 可读性、JD 对齐、证据卡片和面试承接做得更可信。

如果你愿意试用，我最希望你给我的反馈是：

1. 哪些判断明显不准
2. 哪些建议真的帮助你改简历
3. 你还希望它下一步补什么

适合人群：

- 正在投递的应届生 / 社招求职者
- 愿意提供脱敏简历和目标 JD 做测试

### 4. 小红书发帖文案

标题 1：

`很多简历诊断工具都不可信，我决定自己做一个`

标题 2：

`我做了个简历诊断系统，不只打分，还告诉你为什么`

正文模板：

最近在做一个中文求职场景下的简历诊断系统。

我最不满意很多工具的一点是：几秒钟直接给你一个分，但你根本不知道它为什么这么判断，也不知道下一步到底该改哪。

所以我现在做的版本重点是：

1. 看 ATS 能不能顺利读懂你的简历
2. 看你的经历是不是真的对齐目标 JD
3. 把每条关键要求拆成证据卡片
4. 再根据当前缺口生成更像真实面试的追问

这版还加了公开资料补充，会参考公开岗位要求、面经和技能资料，但不会直接覆盖分数。

如果你正在找工作，愿意帮我试用并吐槽，我可以继续把它改得更靠谱。

图片顺序：

1. 封面图：不要只看分数，要看证据
2. 首页上传页
3. 结果页总览
4. 要求级证据卡片
5. 面试承接
6. 公开资料补充

CTA：

`如果你愿意试用，可以看我主页 GitHub / 私信我你的脱敏简历和目标 JD。`

### 5. 知乎长文文案大纲

标题建议：

- 为什么很多简历诊断工具不可信？
- ATS 到底在看什么？我做了一个更可解释的简历诊断系统

文章结构：

1. 市面上常见问题：只给分、不解释、不可复核
2. 我理解的求职者真实痛点
3. 这个项目的判断机制
4. 为什么要把公开资料补充和面试承接接进来
5. 几个真实样例截图
6. 欢迎试用和反馈

### 6. B 站视频脚本

标题：

`我做了个更可信的简历诊断系统：不只给分，还告诉你为什么`

30-60 秒脚本：

1. 开头 5 秒：很多简历工具几秒打分，但你不知道它为什么这么判
2. 10 秒演示上传简历和 JD
3. 15 秒展示结果页顶部：ATS / 匹配度 / 置信度
4. 10 秒展示证据卡片
5. 10 秒展示面试承接
6. 结尾：如果你正在找工作，欢迎试用和吐槽

### 7. 掘金发帖方向

适合发开发日志和技术拆解：

- 我如何把单体 Streamlit 原型拆成 FastAPI 架构
- 如何做一个可解释的 ATS + JD 匹配引擎
- 为什么简历诊断不能只靠一次大模型输出
- 我如何给简历诊断结果加入公开资料补充

### 8. V2EX 发帖方向

更适合发“求反馈”而不是“求下载”。

标题建议：

`[求反馈] 我在做一个面向中文求职者的简历诊断工具，想验证它到底有没有用`

正文建议：

- 先讲动机
- 再讲现在能做什么
- 最后讲你最想听到的反馈是什么

## 五、执行顺序

建议按这个顺序发：

1. 先更新 GitHub README 和截图
2. 再发 GitHub Beta 招募 Issue
3. 再发小红书和知乎
4. 同步发一条掘金开发日志
5. 最后去 V2EX 求反馈

## 六、非常重要的隐私提醒

- 不要发布真实姓名、手机号、邮箱完整信息
- 不要把用户原始 PDF 简历直接上传到仓库
- 所有案例图都尽量脱敏
- 发图前先跑一次 `scripts/check_publish_readiness.ps1`
