# Resume Navigator 大模型接入决策

版本：v0.2  
日期：2026-04-22

## 1. 核心结论

这个项目应该接入大模型，但不应该把大模型放在“拍脑袋打总分”的核心位置。

最合理的架构是：

1. 确定性规则层负责评分和证据
2. 大模型层负责解释、改写、对话和个性化建议

## 2. 为什么不能让大模型直接负责总分

如果让大模型直接给总分，问题会很明显：

1. 同一份简历重复分析，结果容易漂移
2. 用户不知道它为什么这么判
3. 一旦判断错，很难定位是哪里错
4. 成本会被每次分析直接拉高
5. 后续做测试和回归会非常痛苦

## 3. 为什么又必须保留大模型

因为有些任务确定性规则并不擅长：

1. 把问题解释成人话
2. 按目标岗位重写 bullet
3. 和用户对话式地澄清经历
4. 生成模拟面试问题和回答框架
5. 结合用户风格给出多版本表达

## 4. 最适合这个项目的大模型职责

### 适合交给大模型

1. 分析结果解释
2. 按 JD 改写简历 bullet
3. 生成“更职业但不夸张”的版本
4. 面试问题生成
5. 面试回答结构建议

### 不适合完全交给大模型

1. ATS 风险判断
2. 页面数、表格、图片、提取质量判断
3. section 识别
4. 硬技能覆盖统计
5. must-have 要求覆盖统计

## 5. 我对你这个项目的建议

短期建议：

- 先把确定性内核做稳
- 再加“可选的大模型增强层”

更具体地说：

### 第一阶段

只保留现在的确定性分析主链：

- 简历解析
- ATS 规则
- JD 解析
- 匹配评分
- 结果解释页面

### 第二阶段

增加可选的大模型能力，但默认不强依赖：

- 结果解释增强
- 简历 bullet 改写
- 改写前后对比
- 面试问题生成

### 第三阶段

再考虑更深入的对话式产品：

- 追问用户缺失信息
- 帮用户把项目经历写完整
- 生成岗位定制版简历草稿

## 6. 从产品角度看，大模型应该怎样呈现

不要把按钮写成：

- “AI 打分”
- “AI 一键优化”

更好的表达应该是：

- “生成更可执行的改写建议”
- “按目标 JD 改写这段经历”
- “把这条经历改成更适合投递的版本”
- “生成该岗位高概率面试问题”

这样产品会更可信，也更不容易显得虚。

## 7. 从工程角度看，推荐的实现方式

建议做成 Provider 抽象，而不是把某个模型写死在业务代码里。

建议结构：

1. `backend/providers/llm/base.py`
2. `backend/providers/llm/openai_provider.py`
3. `backend/providers/llm/gemini_provider.py`
4. `backend/services/rewrite/`
5. `backend/services/interview/`

这样以后你可以：

- 切换模型
- 对比输出质量
- 控制成本
- 在没有 key 的情况下保留基础功能

## 8. 成本控制建议

这是你项目非常重要的一点。

推荐策略：

1. 默认分析走确定性规则
2. 只有在用户点击“生成改写”时才调大模型
3. 优先用更便宜、更快的模型做解释和改写初稿
4. 对长文本做分段，而不是整份简历一次性重写

## 9. 原创性怎么保证

原创性不在于“完全不用别人都用的大模型”，而在于：

1. 你的评分体系是你自己的
2. 你的证据卡片是你自己的
3. 你的改写规则是你自己的
4. 你的中文求职场景理解是你自己的
5. 你的“禁止胡编”约束是你自己的

所以我的建议是：

接入大模型，但把产品灵魂放在你自己的规则体系和产品设计上。
