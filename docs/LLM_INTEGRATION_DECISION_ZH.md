# Resume Navigator 大模型接入决策
版本：v0.3  
日期：2026-04-22

## 核心结论

这个项目应该接入大模型，但不应该让大模型负责“拍脑袋打总分”。

最合理的结构是：

1. 确定性规则层负责评分、证据、ATS 风险判断。
2. 大模型层负责解释、改写、追问、面试承接。

## 为什么不能让大模型直接打总分

如果把最终判断完全交给大模型，会出现 5 个问题：

1. 同一份简历重复分析，结果容易漂移。
2. 用户很难知道它为什么这么判。
3. 一旦误判，很难定位到底是哪一层出了问题。
4. 每次分析都会直接增加成本。
5. 后续做测试、回归和灰度会非常痛苦。

## 为什么又必须保留大模型

因为有一些任务确定性规则并不擅长：

1. 把结果解释成人话。
2. 按目标 JD 重写某一段经历。
3. 追问用户缺失信息并补足表达。
4. 生成高质量、岗位相关的面试问题。
5. 提供多版本表达而不是只给一个死答案。

## 对这个项目最适合的大模型职责

适合交给大模型：

1. 结果解释增强。
2. 按 JD 改写简历 bullet。
3. 生成“更职业但不夸张”的版本。
4. 面试问题生成。
5. 面试回答框架建议。

不适合完全交给大模型：

1. ATS 风险判断。
2. 页数、表格、图片、文本提取质量判断。
3. section 识别。
4. 硬技能覆盖统计。
5. must-have 要求覆盖统计。

## 推荐落地方式

第一阶段：

- 先把确定性分析内核做稳。
- 让无模型 key 的用户也能跑完整诊断。

第二阶段：

- 增加可选的大模型解释与改写。
- 默认不强依赖大模型。

第三阶段：

- 做对话式补充信息。
- 做岗位定制版改写和面试承接。

## 工程建议

建议走 Provider 抽象，而不是把某个模型写死在业务代码里。

推荐目录：

1. `backend/providers/llm/base.py`
2. `backend/providers/llm/openai_provider.py`
3. `backend/providers/llm/gemini_provider.py`
4. `backend/services/rewrite/`
5. `backend/services/interview/`

## 成本控制建议

1. 默认分析只走确定性引擎。
2. 只有在用户点击“生成改写”或“生成面试问题”时才调用大模型。
3. 优先使用便宜模型做首轮解释和改写。
4. 长文本按段调用，避免整份简历一次性重写。

## 原创性怎么保证

原创性不在于“完全不用别人也在用的大模型”，而在于：

1. 你的评分体系是自己的。
2. 你的证据卡片机制是自己的。
3. 你的中文求职场景约束是自己的。
4. 你的反造假规则是自己的。
5. 你的简历到面试闭环设计是自己的。
