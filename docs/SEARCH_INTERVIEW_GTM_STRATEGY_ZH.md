# Resume Navigator 搜索、面试承接与推广执行策略
版本：v0.3  
日期：2026-04-22

## 1. 搜索功能要不要保留

要保留，但只能作为“证据增强层”，不能继续做成“评分核心”。

正确定位：

1. 为岗位补充行业通用偏好。
2. 为用户提供来源可追溯的证据卡片。
3. 为面试承接提供岗位背景素材。

不应该继续保留的做法：

1. 把其他公司内容替换成目标公司名。
2. 用未经校验的社区帖子直接生成结论。
3. 让搜索结果直接决定总分。

推荐改法：

1. 每条搜索洞察必须带来源 URL、抓取时间、来源类型、置信度。
2. 搜索层默认关闭，只有用户主动开启或本地研究模式才启用。
3. 搜索结果只参与“补充解释”和“建议优先级”，不直接改写确定性分数。

## 2. “面试承接”到底指什么

这里不是泛泛的聊天式面试模拟，而是“从简历诊断自然延伸到高质量面试准备”。

重点应该抓住 4 个真实痛点：

1. 用户知道自己哪里弱，但不知道会怎么被问。
2. 用户会写经历，但不会讲经历。
3. 用户能回答事实，却不会组织成面试语言。
4. 用户不知道 JD 的 must-have 会怎么落到面试里。

所以本项目里的“面试承接”建议拆成 3 层：

1. 弱项追问：
   从本次诊断里抓出最可能被问的缺口。
2. 问题生成：
   生成高概率问题，而不是泛泛题库。
3. 回答框架：
   给出 STAR / 项目复盘 / 指标表达框架，帮助用户说清楚。

## 3. 推荐的产品路线

简历到面试的闭环建议做成：

1. 上传简历和 JD。
2. 先做确定性诊断。
3. 生成要求级证据卡片。
4. 用户选择“按 JD 改写”。
5. 系统生成高概率面试问题。
6. 系统提示哪些问题最可能卡住用户。
7. 用户按问题补强简历与答案。

## 4. 推广怎么做

这个项目最适合“边开发边推广”，不适合闭门做完再发。

原因：

1. 求职产品的真实价值高度依赖具体案例。
2. 负反馈比空泛点赞更值钱。
3. 冷启动阶段最重要的是拿到真实简历 + JD 组合。

推荐节奏：

### 阶段 1：GitHub 冷启动

1. 每周至少一次可见提交。
2. 每周一条开发日志。
3. 每次更新都明确写出“这次修了什么误判”。

### 阶段 2：中文社区招募

1. 发“求反馈帖”，不是发广告帖。
2. 强调“欢迎指出误判”。
3. 用前后对比和误判修复案例吸引试用者。

### 阶段 3：案例驱动扩散

1. 发布匿名化案例。
2. 展示诊断前后差异。
3. 展示真实用户反馈如何影响迭代。

## 5. 哪些事情我可以直接接管，哪些不行

我现在可以直接接管：

1. 仓库代码开发。
2. 本地验证和测试脚本。
3. Git 提交与推送。
4. README、文档、发布素材、招募文案编写。
5. 产品功能与页面迭代。

我现在不能直接替你完成的外部动作：

1. 在需要登录的平台发帖。
2. 私信真实用户招募。
3. 使用你的社交平台账号互动。

如果你希望我继续“更强接管”，你需要提供的是：

1. 对应平台的已登录会话或可用登录方式。
2. 明确授权我代你发布和回复。

## 6. 试用者来源怎么选

建议优先从这三类人里招募：

1. 正在投递的应届生。
2. 近 3 个月想转岗的人。
3. 已经有明确 JD、但简历通过率不高的人。

不建议的做法：

1. 未经授权抓取个人简历并分析。
2. 把网上公开简历当成产品案例直接使用。
3. 在没有匿名化处理时公开展示真实求职材料。

更合理的替代方案：

1. 用公开、匿名、允许引用的简历样例。
2. 用用户主动提交的简历。
3. 对所有案例做脱敏展示。
