# Resume Navigator 前端重构方案

版本：V0.2  
日期：2026-04-22

## 1. 这次重构要解决什么

这次前端不是单纯“做得更好看”，而是要解决 4 个核心问题：

1. 用户进入首页后，要在 5 秒内知道这个产品能帮自己做什么。
2. 结果页不能再像“秒出一个分数”，而要让人感觉这是经过多步复核后的判断。
3. 用户看完结果后，要知道下一步该做什么，而不是停留在“知道问题”。
4. 开发者信息、版本信息、方法说明可以保留，但必须退到次要层，不抢主任务入口。

## 2. 设计方向

整体方向：

- 中文优先，英文作为切换能力
- 专业、克制、可信，不做花哨 AI 工具站
- 信息密度高，但层次清楚
- 先证据，后结论，再行动

明确不要做成：

- 首页大段宣传文案堆砌
- 满屏“AI 智能优化”“一键秒出结果”的廉价感表达
- 开发者说明和用户主操作入口混在一起

## 3. 信息架构

### 首页

首页只保留 3 个主任务：

1. 上传简历
2. 粘贴 JD
3. 选择分析模式并开始分析

首页应该重点呈现的信息：

- 产品一句话价值
- 可信分析链路的简短说明
- 上传区和 JD 输入区
- 深度分析模式的说明
- 历史记录和反馈入口

首页不应该大面积占空间的信息：

- 长篇开发日志
- 过多路线图
- 太多技术实现细节

### 结果页

结果页改成 5 个主区域：

1. 顶部总览
2. 为什么这个结果可信
3. 证据与缺口
4. 下一步动作
5. 面试承接

### 历史页

历史页只保留最关键字段：

- 时间
- 目标岗位
- ATS
- 匹配度
- 可信度
- 查看结果入口

### 反馈页

反馈页重点展示：

- 用户觉得哪里不准
- 用户最想要什么功能
- 反馈对应的是哪次分析

## 4. 首页布局方案

### 顶部导航

- 左侧：品牌名
- 右侧：历史、反馈、API 文档、语言切换

### 首屏布局

左侧：

- 核心标题
- 一句话副标题
- 三个可信能力卡片
- 分析量 / 反馈量 / 方法论提示
- 折叠式开发说明

右侧：

- 上传简历
- 粘贴 JD
- 分析模式切换
- 开始分析按钮

### 首屏下方

保留两个次要信息块：

- 这次会给用户什么
- 开发阶段的推广与试用原则

## 5. 结果页布局方案

### 模块 A：顶部总览

展示：

- 文件名
- 目标岗位
- ATS 分数
- 岗位匹配度
- 判断可信度
- 当前模式

要求：

- 分数卡片足够醒目
- 标题和结论短，不要大段解释
- 操作按钮放在第一屏就能看到

### 模块 B：为什么这次判断可信

这是结果页最关键的信任模块，要解决“出分太快、不像认真分析”的问题。

展示内容：

- 分析阶段链路
- 每一步的说明
- 每一步的耗时
- 当前可信度和可信原因

要求：

- 不要人为加很长假延迟
- 但要把真实分析链路显式展示出来
- 深度模式必须体现出额外复核

### 模块 C：证据与缺口

左侧重点：

- 优势信号
- 风险信号
- 证据高亮

右侧重点：

- JD 要求证据卡
- 技能覆盖
- 缺失关键词

要求：

- 所有“缺口”都尽量对应到具体要求
- 所有“建议”都尽量能落到具体修改动作

### 模块 D：下一步动作

这里不再只是列问题，而要明确 CTA：

1. 优先修什么
2. 为什么优先修这个
3. 修到什么程度才算过关

### 模块 E：面试承接

这里不是泛泛面试模拟，而是：

- 根据岗位家族生成更像真实一面的题
- 根据简历缺口生成核验题
- 根据 JD 要求生成证明题

每张卡片展示：

- 问题
- 为什么会被问
- 回答重点
- 优先级

## 6. 开发者信息如何保留

开发者信息要保留，但必须弱化展示。

推荐方式：

- 折叠式 disclosure
- 轻量 badge
- tooltip / 小提示语
- 右侧次要信息卡

适合放进去的信息：

- 当前版本说明
- 分析方法说明
- 数据来源策略说明
- 当前是否启用深度复核

不适合直接铺开的信息：

- 大段开发日志
- Prompt 细节
- 内部调试数据

## 7. 文案原则

前端文案顺序：

1. 先说当前判断
2. 再说判断依据
3. 最后说下一步动作

避免使用：

- “AI 一键秒出”
- “全面提升求职竞争力”
- “智能优化你的未来”

更好的表达：

- “当前最大的缺口是某项要求没有证据支撑”
- “这次判断可信度高，因为 JD 和简历都具备足够结构化信息”
- “建议先补证据，再做措辞优化”

## 8. 状态设计

### 加载态

不要只显示 spinner。

建议明确展示：

1. 正在解析简历
2. 正在提取 JD 要求
3. 正在审核 ATS 风险
4. 正在逐条比对 JD
5. 正在生成下一步动作和面试问题

### 错误态

错误信息要明确区分：

- 文件问题
- JD 过短或过弱
- 解析失败
- 服务异常

### 空状态

历史页和反馈页都要有清晰空状态，不要留空白。

## 9. 移动端要求

移动端优先级：

1. 上传和输入必须可用
2. 结果页分数卡片优先显示
3. 右侧边栏内容自动下沉
4. 面试题卡不要过长

## 10. 执行顺序

### Phase 1

- 首页信息收敛
- 结果页重排
- 开发者信息弱化
- 加入明确加载态

### Phase 2

- 做更成熟的对比视图
- 增加按 JD 改写入口
- 增强历史页过滤与对比

### Phase 3

- 做移动端细节打磨
- 做结果分享视图
- 做更成熟的用户反馈闭环
