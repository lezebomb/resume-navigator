# Resume Navigator 开发期增长与推广手册

版本：v0.1
日期：2026-04-22
状态：可执行草案

## 1. 核心结论

这个产品不应该等“全部做完”再推广。

更适合你的打法是：

- 一边开发
- 一边公开记录构建过程
- 一边收真实简历和真实 JD
- 一边打磨产品定位
- 一边建立第一批种子用户和贡献者

对你这种项目来说，开发阶段本身就是推广阶段。

## 2. 你这个产品最适合怎么定位

不建议对外说：

- AI 简历优化神器
- 一键润色拿 offer
- 超强 ATS 打分工具

建议对外说：

- 中文求职场景下的简历诊断与改写辅助系统
- 强调 ATS 可读性、JD 对齐和可解释建议的求职助手
- 不乱编经历、可本地部署、适合真实求职反馈闭环的开源项目

## 3. 为什么开发阶段就要开始推广

因为你现在最缺的不是“曝光”，而是：

- 真实用户问题
- 真实简历样本
- 真实 JD 样本
- 真实负反馈
- 第一批愿意帮你传播的人

如果你等全部做完再推广，你很容易遇到三个问题：

- 做了很多功能，但用户并不在乎
- 定位太宽，别人听不懂你做什么
- 没有首批案例和口碑，发布时声音很小

## 4. 开发期增长目标

### 第一个月

- 找到 10 到 20 个真实求职者试用
- 收集至少 30 份真实 JD 样本
- 收集至少 10 份匿名简历样本
- 明确 3 个最有价值的核心场景

### 第二个月

- 建立稳定的内容更新节奏
- 做出 3 到 5 个前后对比案例
- 获得第一批公开反馈或 issue
- 让 GitHub 仓库具备“有人愿意 star 和 follow”的样子

### 第三个月

- 建立早期用户群或种子测试群
- 形成可复用的演示链路
- 开始尝试开源社区和产品社区的集中曝光

## 5. 最适合你的推广渠道

### 第一优先级：GitHub

你的 GitHub 仓库本身就是产品名片。

要点：

- 让 README 一眼看懂价值
- 持续提交，保持活跃记录
- 用 issue 收真实需求
- 后续可开启 Discussions 收社区反馈

参考：

- [GitHub community health files](https://docs.github.com/en/communities/setting-up-your-project-for-healthy-contributions/creating-a-default-community-health-file)
- [GitHub Discussions](https://docs.github.com/en/repositories/managing-your-repositorys-settings-and-features/enabling-features-for-your-repository/enabling-or-disabling-github-discussions-for-a-repository)

### 第二优先级：中文开发者内容社区

适合发布：

- 开发日志
- 架构思考
- ATS 判断逻辑
- 简历诊断案例拆解

建议平台：

- [掘金](https://juejin.cn/)
- [V2EX](https://www.v2ex.com/)

### 第三优先级：产品曝光平台

适合在有演示链接、截图、视频和核心价值说明后集中发布。

建议平台：

- [Product Hunt](https://www.producthunt.com/)

### 第四优先级：求职人群聚集地

适合做真实用户增长，不适合一上来发很硬的“推广帖”。

更适合的打法：

- 发真实案例
- 发踩坑总结
- 发“怎么判断你的简历为什么过不了 ATS”
- 顺带带出项目

适合的载体：

- 你的朋友圈/微信群
- 校友群、求职群、实习群
- 小红书、知乎、B站上的案例内容

## 6. 你的增长飞轮应该长什么样

```text
真实用户问题
  -> 做出一个具体改进
  -> 写成案例或开发日志
  -> 吸引更多相关用户
  -> 获得更多简历/JD/反馈
  -> 产品更准
  -> 再产出更强案例
```

## 7. 开发阶段就能做的 8 件事

1. 每周发一条“本周构建进展”
2. 每周找 3 到 5 个真实求职者试用
3. 每周整理 1 个“简历诊断前后对比案例”
4. 每周写 1 条“ATS / JD 对齐 / 简历表达”的短内容
5. 用 GitHub issue 代替散乱聊天记录
6. 保留用户原话，后面做 landing page 证据
7. 建一个种子测试名单
8. 为每个新增功能都准备一句外行能听懂的话

## 8. 你现在的最佳内容策略

不要一上来就讲“大模型架构”。

先讲用户真正有感知的东西：

- 为什么很多简历不是“能力不行”，而是 ATS 根本没读对
- 为什么同一份简历投不同 JD，结果会差很多
- 为什么“AI 一键润色”会有造假风险
- 为什么“可解释建议”比“总分”更重要

## 9. 推荐的内容结构模板

### 模板 A：问题拆解型

- 一个真实问题
- 传统做法为什么不够
- 你的系统怎么判断
- 真实结果长什么样
- 你下一步还要优化什么

### 模板 B：开发日志型

- 本周做了什么
- 为什么做这个
- 真实测试发现了什么
- 下周准备做什么

### 模板 C：案例对比型

- 某类 JD 的要求
- 某类简历常见问题
- 系统给出的高优先级建议
- 修改后的变化

## 10. 适合你现在就发的内容题目

- 我在做一个不乱编经历的简历诊断系统，先把 ATS 读懂这件事做好
- 为什么很多简历不是写得差，而是结构让 ATS 读错了
- 我把简历诊断从 AI 打分器改成了确定性规则 + 可解释建议
- 做中文求职产品时，为什么不能把一切都交给大模型
- 我在重构一个简历项目：保留行业洞察，但不把高风险抓取做成公开依赖

## 11. 你应该重点积累什么素材

- 真实匿名案例
- 产品界面截图
- 真实测试输出结果
- 用户原话反馈
- 每周 changelog
- 路线图
- 演示视频

## 12. 开发阶段的转化路径

### 目标不是立即收费

目标是先完成：

- 看懂
- 感兴趣
- 愿意试
- 愿意反馈
- 愿意转发

### 最小转化漏斗

- 内容帖
- GitHub 仓库或演示页
- 试用
- 反馈表单或 issue
- 测试群或邮件列表

## 13. 一周执行节奏建议

### 周一

- 规划本周要做的一个核心改进

### 周二到周四

- 开发
- 每天记录 1 条可发的构建素材

### 周五

- 发一条开发日志
- 找 2 到 3 位用户试用

### 周末

- 整理案例
- 复盘哪些功能真正引起兴趣

## 14. 对你最重要的早期指标

- 每周新增试用人数
- 每周新增真实 JD 数量
- 每周新增匿名简历样本数
- 每周新增 issue / 反馈数
- 内容帖的收藏和私信数量
- 从内容进入 GitHub 的点击量

## 15. 你现在不该做的推广动作

- 过早投广告
- 做太重的品牌包装
- 写一堆空泛宣传文案
- 追求一次性大爆发
- 在没有演示和案例时去大规模发布

## 16. 适合你现在就落地的动作清单

1. 补齐 GitHub 仓库基础文件
2. 每周至少发 1 条开发进展
3. 建一个“种子用户反馈收集表”
4. 让每次版本更新都能变成一条内容
5. 开始收集匿名案例素材
6. 准备首个 60 秒演示视频脚本

## 17. 60 秒演示视频脚本模板

- 5 秒：一句话讲清产品做什么
- 10 秒：展示输入，一份简历 + 一份 JD
- 20 秒：展示 ATS 风险和匹配结果
- 15 秒：展示优先修改动作
- 10 秒：讲清产品与“一键 AI 润色”的区别

## 18. 最适合你的开发期推广定位句

你可以直接对外用这句：

“我在做一个中文求职场景下的简历诊断系统，重点不是一键生成，而是让 ATS 可读性、JD 对齐和修改建议更真实、更可解释。”
